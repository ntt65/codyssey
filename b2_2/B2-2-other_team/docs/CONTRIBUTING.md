# 협업 가이드

팀 과제 시작 후 협업 규칙을 이곳에 정리합니다.

## Github Flow를 선택한 이유

### 1. 지속적 배포(CD)와 빠른 피드백 루프
* GitHub Flow는 **"main 브랜치는 항상 배포 가능한 상태여야 한다"**는 단 하나의 엄격한 규칙을 기반으로 움직입니다.
* 브랜치 구조가 단순하여 기능을 개발한 즉시 `main`에 병합하고 프로덕션 환경에 배포할 수 있습니다. 
* 이를 통해 사용자 및 팀원들로부터 빠르게 피드백을 받아 서비스를 민첩하게 개선할 수 있습니다.

### 2. 낮은 복잡성과 협업 비용 감소
* `develop`, `release`, `hotfix` 등 수많은 관리용 브랜치를 두는 Git Flow와 달리, GitHub Flow는 `main`과 `feature(작업 브랜치)`로만 구성됩니다.
* 브랜치 관리 및 동기화에 드는 오버헤드가 적어, 팀원들이 브랜치 전략 자체를 익히는 데 드는 비용을 줄이고 오직 **코드 구현과 코드 리뷰에만 집중**할 수 있습니다.

### 3. Pull Request(PR) 중심의 비동기 교차 검증
* 모든 변경 사항은 `main`에 직접 push할 수 없으며, 반드시 PR을 거쳐야 합니다.
* PR을 통해 팀원 간 자연스러운 코드 리뷰(Peer Review)가 활성화되며, 이는 코드 품질 향상과 더불어 팀 전체가 시스템 구조를 공동으로 책임지는 문화를 만듭니다.
* 수십 개의 지저분한 작업 커밋이 있더라도 최종 병합 시 `Squash and Merge`를 활용하여 `main` 히스토리를 언제나 깔끔하고 추적 가능하게 유지할 수 있습니다.

| 비교 항목 | Git Flow (전통적/체계적) | GitHub Flow (현대적/애자일) |
| :--- | :--- | :--- |
| **핵심 철학** | "완벽하게 검증하고 정기적으로 출시한다" | "수정했으면 리뷰받고 즉시 배포한다" |
| **브랜치 구조** | **복잡함 (5종류)**<br>`main`, `develop`, `feature`, `release`, `hotfix` | **단순함 (2종류)**<br>`main`, `작업 브랜치(feature/fix 등)` |
| **중심 브랜치** | `main` (배포용) + `develop` (개발 통합용) | `main` (배포 및 개발 기준점) |
| **배포 주기** | 정기 배포 (주 단위, 월 단위, 버전별) | 수시 배포 (하루에도 수십 번 가능, CD 필수) |
| **코드 리뷰 시점** | 배포 직전 혹은 대규모 머지 시점 | 모든 작은 작업 단위(PR)마다 상시 진행 |
| **주요 장점** | 버전을 엄격히 통제하여 대규모 안정성 확보 | 브랜치 관리 비용이 적고 피드백 루프가 빠름 |
| **주요 단점** | 브랜치 동기화 및 머지 충돌(Conflict) 비용 높음 | 테스트 자동화(CI/CD)가 없으면 main이 쉽게 오염됨 |
| **추천 프로젝트** | 모바일 앱(스토어 심사 필수), 패키지 SW, 금융 시스템 | 웹 서비스, SaaS, 스타트업 및 소규모 팀 프로젝트 |

## 브랜치 네이밍 규칙

브랜치명은 **작업 유형/이슈 번호** 형식으로 작성한다.

```bash
작업유형/이슈번호
```

### 예시

```bash
docs/2
feat/3
fix/4
refactor/5
chore/6
```

### 작업 유형

| 유형       | 설명                                 |
| ---------- | ------------------------------------ |
| `feat`     | 새로운 기능 추가                     |
| `fix`      | 버그 수정                            |
| `docs`     | 문서 작성 및 수정                    |
| `refactor` | 기능 변경 없는 코드 구조 개선        |
| `chore`    | 설정, 빌드, 패키지 관리 등 기타 작업 |

### 작성 규칙

- 브랜치명은 소문자로 작성한다.
- 공백은 사용하지 않는다.
- 이슈 번호 앞에는 `#`을 붙이지 않는다.
- `/`를 기준으로 작업 유형과 이슈 번호를 구분한다.
- 하나의 브랜치는 하나의 이슈를 기준으로 생성한다.

## PR 작성 규칙

- 모든 작업은 Issue 생성 후 진행한다.
- 작업 완료 후 `feature/*` 브랜치에서 PR을 생성한다.
- 하나의 PR에는 하나의 작업만 포함한다.
- 관련 없는 파일 수정은 포함하지 않는다.
- `main` 브랜치로 직접 push하지 않는다.

## 목적

PR(Pull Request)은 작업 내용을 팀원에게 공유하고 코드 리뷰를 받기 위한 절차이다. 모든 변경 사항은 PR을 통해 `main` 브랜치에 반영한다.

## PR 제목 규칙

PR 제목은 커밋 메시지 컨벤션과 동일한 형식을 사용한다.

예시:

```text
feat: 문자열 길이 검증 함수 추가
docs: PR 작성 규칙 작성
fix: 길이 검증 함수 예외 처리 수정
```

## PR 본문 필수 항목

모든 PR 본문에는 아래 내용을 포함해야 한다.

### 연결 이슈

```text
Closes #이슈번호
```

### 변경 사항 (What)

- 어떤 내용을 추가하거나 수정했는지 작성

### 변경 이유 (Why)

- 해당 변경이 필요한 이유 작성

### 테스트/검증 방법 (How)

- 변경 사항을 어떻게 확인했는지 작성

예시:

```text
## 연결 이슈
Closes #7

## 변경 사항
- validate_length 함수 추가

## 변경 이유
- 문자열 길이 검증 기능 제공

## 테스트/검증 방법
- 최소 길이, 최대 길이 조건 테스트 완료
```

## 리뷰 반영 규칙

- 리뷰 의견을 확인한 후 답글을 작성한다.
- 필요한 경우 수정 커밋을 추가한다.
- 리뷰 반영 후 변경 내용을 PR에 남긴다.
- 최소 1명 이상의 승인 후 병합한다.

## 병합 규칙

- 승인 없이 병합하지 않는다.
- 충돌 발생 시 해결 후 다시 리뷰를 요청한다.
- 병합 후 연결된 Issue가 정상적으로 종료되었는지 확인한다.

## 코드 리뷰 규칙

- 모든 PR은 작성자를 제외한 팀원에게 리뷰를 요청한다.
- 리뷰어는 PR의 변경 내용이 연결된 Issue의 작업 범위와 일치하는지 확인한다.
- 리뷰어는 코드 또는 문서가 팀의 브랜치 규칙, 커밋 메시지 규칙, PR 작성 규칙을 따르는지 확인한다.
- 리뷰 코멘트는 `LGTM`, `좋아요`, `확인했습니다`만 단독으로 작성하지 않는다.
- 리뷰어는 파일 또는 라인을 기준으로 질문, 대안 제안, 리스크 지적, 개선 제안 중 1개 이상을 작성한다.
- PR 작성자는 리뷰 코멘트에 최소 1회 이상 답글을 남기거나, 수정 커밋을 통해 리뷰 내용을 반영한다.
- 리뷰를 반영한 경우 어떤 부분을 수정했는지 PR 코멘트로 간단히 남긴다.
- 리뷰 반영 내역과 관련 PR 링크는 최종 제출 시 `SUBMISSION.md`에 정리한다.

### 리뷰어 확인 항목

- 변경 내용이 Issue의 목적과 일치하는가?
- 불필요한 파일이나 작업 범위 밖의 변경이 포함되지 않았는가?
- 함수명, 변수명, 파일명이 역할을 이해하기 쉽게 작성되었는가?
- 예외 상황이나 빈 값 처리 기준이 필요한 경우 명확히 작성되었는가?
- 문서 변경의 경우 문장이 이해하기 쉽고 팀 규칙으로 사용하기에 충분한가?
- PR 본문에 변경 사항, 변경 이유, 검증 방법이 작성되어 있는가?

### 리뷰 코멘트 예시

```text
src/data_utils.py의 평균 계산 함수에서 빈 리스트가 들어오는 경우의 처리 기준이 필요해 보입니다.
빈 리스트 입력 시 0을 반환할지, ValueError를 발생시킬지 docstring에 명시하면 좋겠습니다.
```

```text
docs/CONTRIBUTING.md의 코드 리뷰 규칙이 명확합니다.
다만 리뷰 반영 내역을 SUBMISSION.md에 정리한다는 기준을 추가하면 제출 증빙 관리에 더 도움이 될 것 같습니다.
```

### PR 작성자 응답 예시

```text
좋은 의견 감사합니다. 빈 리스트 입력 시 ValueError가 발생하도록 처리하고, docstring에 예외 조건을 추가했습니다.
```

## 충돌 대응 흐름

- PR에서 충돌이 발생했거나, 작업 브랜치에 최신 `main`을 반영하는 과정에서 충돌이 발생하면 PR 작성자가 우선 해결한다.
- 충돌 해결 전, 현재 작업 내용을 커밋하거나 저장한다.
- 로컬 `main` 브랜치를 최신 상태로 만든 뒤, 작업 브랜치에 병합한다.

1. `main` 브랜치로 이동한다.
   ```bash
   git checkout main
   ```
2. `main` 브랜치의 최신 내용을 가져온다.
   ```bash
   git pull origin main
   ```
3. 충돌이 발생한 작업 브랜치로 이동한다.
   ```bash
   git checkout 작업 브랜치명
   ```
4. `main` 브랜치의 최신 내용을 작업 브랜치에 반영한다.
   ```bash
   git merge main
   ```
5. 충돌이 발생한 파일을 열어 충돌 표시를 확인한다.
   ```text
   <<<<<<< HEAD
   현재 작업 브랜치의 내용
   =======
   main 브랜치에서 가져온 내용
   >>>>>>> main
   ```
6. 팀원과 상의가 필요한 부분은 임의로 삭제하지 않고 확인 후 결정한다.
7. 충돌 내용을 정리한 뒤 충돌 표시를 모두 제거한다.
8. 수정한 파일을 다시 추가하고 커밋한다.
   ```bash
   git add .
   git commit -m "fix: merge conflict 해결"
   ```
9. 작업 브랜치를 원격 저장소에 다시 push한다.
   ```bash
   git push origin 작업브랜치명
   ```
10. GitHub PR 화면에서 충돌이 해결되었는지 확인한다.
11. 충돌 해결 후에도 변경 내용이 의도한 대로 동작하는지 확인한 뒤 merge한다.
12. 충돌 해결이 어렵거나 다른 팀원의 작업을 수정해야 하는 경우, 해당 팀원과 함께 확인한 뒤 처리한다.

## 커밋 메시지 컨벤션

### 기본 형식

우리 팀은 Git 히스토리에서 각 변경 사항의 목적과 대상을 쉽게 파악할 수 있도록 일관된 커밋 메시지 규칙을 사용한다.

```text
type: 작업 내용 요약
커밋 타입
feat: 새로운 기능 추가
fix: 버그 수정
docs: 문서 작성 또는 수정
refactor: 기능 변경 없이 코드 또는 문서 구조 개선
chore: 설정, 초기 구조, 기타 작업
test: 테스트 코드 추가 또는 수정
좋은 커밋 메시지 예시
feat: 문자열 대소문자 변환 함수 추가
docs: 커밋 메시지 컨벤션 작성
refactor: README 체크리스트 구조 개선
fix: 빈 리스트 평균 계산 예외 처리
chore: 초기 프로젝트 구조 설정
금지하는 커밋 메시지 예시
다음과 같이 변경 대상이나 목적을 알 수 없는 메시지는 사용하지 않는다.
update
fix
temp
wip
final
edit file
bug fix
작성 기준
커밋 메시지는 짧게 작성하되, 무엇을 변경했는지 알 수 있어야 한다.
단순히 작업 상태를 적기보다 변경 대상과 목적이 드러나도록 작성한다.
```
