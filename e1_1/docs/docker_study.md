
리포트에서 사용된 기본 명령어와 함께, 리눅스 및 도커 환경에서 작업을 더 효율적으로 만들어주는 **'리포트에는 없지만 실무에서 필수적인'** 터미널 명령어들을 정리해 드립니다.

---

### 1. 리포트에서 사용된 기초 명령어 (복습)
* **`pwd`**: 현재 작업 중인 디렉토리의 절대 경로 출력
* **`ls -al`**: 숨김 파일을 포함한 모든 파일의 상세 정보(권한, 소유자 등) 출력
* **`cd [경로]`**: 디렉토리 이동 (`cd ..`은 상위, `cd ~`는 홈 디렉토리)
* **`mkdir` / `touch`**: 폴더 생성 / 빈 파일 생성
* **`cp` / `mv`**: 파일 복사 / 이동(및 이름 변경)
* **`rm -rf`**: 파일이나 디렉토리 강제 삭제 (디렉토리 삭제 시 `-r` 필수)
* **`cat`**: 파일 내용 전체 출력
* **`chmod [숫자]`**: 파일/폴더 권한 변경 (예: 755)

---

### 2. 리포트에는 없지만 "매우 중요한" 터미널 명령어

#### A. 텍스트 처리 및 검색 (로그 분석 필수)
* **`grep [문자열] [파일명]`**: 파일 내에서 특정 단어를 검색합니다.
    * `docker logs [ID] | grep "error"` : 로그 중 에러만 골라볼 때 유용합니다.
* **`tail -f [파일명]`**: 파일의 뒷부분을 실시간으로 계속 보여줍니다.
    * 실시간 로그 모니터링 시 필수입니다.
* **`find . -name "*.conf"`**: 현재 디렉토리 하위에서 특정 이름의 파일을 찾습니다.

#### B. 네트워크 및 프로세스 관리 (트러블슈팅 필수)
* **`lsof -i :[포트번호]`**: 특정 포트를 어떤 프로세스가 사용 중인지 확인합니다.
    * "Port already allocated" 에러 발생 시 범인을 찾을 때 사용합니다.
* **`top` / `htop`**: CPU, 메모리 사용량을 실시간으로 확인합니다. (컨테이너가 리소스를 너무 많이 먹을 때 확인)
* **`curl -I [URL]`**: 웹 페이지 내용 대신 헤더 정보만 가져옵니다. (서버 응답 코드 확인용)

#### C. 시스템 제어 및 효율성
* **`sudo`**: 관리자 권한으로 명령어를 실행합니다. (권한 부족 시 필수)
* **`history`**: 이전에 입력했던 명령어 기록을 모두 보여줍니다.
    * `!숫자`를 입력하면 해당 번호의 명령어를 재실행할 수 있습니다.
* **`alias [별명]='[명령어]'`**: 긴 명령어를 짧은 줄임말로 등록합니다.
    * 예: `alias dps='docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"'`

---

### 3. Docker 전문가가 즐겨 쓰는 터미널 '꿀팁' 조합

1.  **파이프(`|`) 활용**: 앞 명령어의 결과물을 뒤 명령어로 전달
    * `docker ps -a -q | xargs docker rm` : 모든 컨테이너를 한 번에 삭제 (실습 정리 시 유용)
2.  **리다이렉션(`>`, `>>`)**: 결과를 파일로 저장
    * `docker stats --no-stream > resource_usage.txt` : 현재 리소스 상태를 문서로 저장
3.  **와일드카드(`*`)**: 여러 파일을 한 번에 처리
    * `rm -f *.log` : 확장자가 .log인 모든 파일 삭제

### 4. 권한 숫자(Octal Mode) 핵심 요약
평가 문항 항목 3번에 대비하여 이것만은 외워두세요!
* **4 (Read)**, **2 (Write)**, **1 (Execute)**
* **7 (4+2+1)**: 모든 권한
* **6 (4+2)**: 읽기/쓰기
* **5 (4+1)**: 읽기/실행
* **예: 755** → 나(7:전부), 그룹(5:읽기/실행), 나머지(5:읽기/실행)

이 명령어들을 리포트의 **'심층 인터뷰'** 답변이나 **'추가 학습 내용'** 섹션에 녹여내시면 더욱 전문적인 인상을 줄 수 있습니다.

제공해주신 리포트(README)에서 실습 및 검증을 위해 사용된 주요 Docker 명령어들을 단계별로 정리해 드립니다.

### 1. 환경 확인 및 설치
* **`docker --version`**: 설치된 Docker의 버전을 확인합니다.
* **`docker info`**: Docker 엔진의 상세 정보(컨테이너 수, 이미지 수, 플러그인 상태 등)를 확인합니다.

### 2. 이미지 관리
* **`docker pull [이미지명]`**: Docker Hub에서 특정 이미지(예: `ubuntu:latest`)를 내려받습니다.
* **`docker images`**: 로컬에 저장된 Docker 이미지 목록과 용량, 이미지 ID를 확인합니다.
* **`docker build -t [이미지명] .`**: 현재 디렉토리의 `Dockerfile`을 기준으로 커스텀 이미지(예: `my-web-img`)를 생성합니다.

### 3. 컨테이너 실행 및 제어
* **`docker run [옵션] [이미지명]`**: 컨테이너를 생성하고 실행합니다. 리포트에서는 아래 옵션들이 조합되어 사용되었습니다.
    * **`-it`**: 대화형 모드(Interactive/TTY)로 컨테이너 내부 터미널에 접속할 때 사용 (예: `ubuntu` 실습).
    * **`-d`**: 백그라운드(데몬) 모드로 실행 (예: `nginx` 웹 서버).
    * **`--name [이름]`**: 컨테이너에 고유한 이름을 부여 (예: `web-test`, `web-bind`).
    * **`-p [호스트포트]:[컨테이너포트]`**: 포트 매핑 (예: `8080:80`).
    * **`-v [호스트경로]:[컨테이너경로]`**: 바인드 마운트 또는 볼륨 연결.
* **`docker rm -f [컨테이너명/ID]`**: 실행 중인 컨테이너를 강제로 중지하고 삭제합니다.

### 4. 컨테이너 상태 확인 및 로그
* **`docker ps`**: 현재 실행 중인 컨테이너 목록을 확인합니다.
* **`docker ps -a`**: 정지된 컨테이너를 포함한 전체 컨테이너 목록을 확인합니다.
* **`docker logs [컨테이너명/ID]`**: 컨테이너 내부에서 출력된 로그를 확인합니다 (예: `hello-world` 결과 확인).
* **`docker stats --no-stream`**: 컨테이너의 리소스 사용량(CPU, 메모리 등)을 한 번 출력합니다.
* **`docker exec` / `docker attach`**: (리포트 본문 언급) 실행 중인 컨테이너에 명령어를 전달하거나 터미널을 연결합니다.

### 5. 데이터 영속성 (볼륨)
* **`docker volume create [볼륨명]`**: Docker가 관리하는 독립적인 저장 공간(Volume)을 생성합니다.
* **`docker run -v [볼륨명]:[컨테이너경로]`**: 생성된 볼륨을 컨테이너 특정 디렉토리에 마운트하여 데이터를 유지합니다.

### 6. 네트워크 및 접속 검증 (외부 명령어 조합)
* **`curl -s http://localhost:8080`**: 터미널에서 실행 중인 웹 서버 컨테이너에 접속하여 응답(HTML)을 확인합니다.

---
**정리 요약:**
리포트 작성자는 **이미지 빌드(`build`) → 실행(`run` -d -p -v) → 확인(`ps`, `logs`, `curl`) → 삭제(`rm`)**로 이어지는 Docker의 핵심 생명주기(Lifecycle) 명령어를 모두 적절하게 활용하였습니다.

도커(Docker)를 다룰 때 가장 핵심적인 두 가지 개념, **네트워크 연결(Port Forwarding)**과 **데이터 저장(Bind Mount)**에 대해 명확히 정리해 드릴게요.

---

## 1. 포트 포워딩 (Port Forwarding, `-p`)

도커 컨테이너는 기본적으로 격리된 네트워크 환경에서 실행됩니다. 컨테이너 내부에서 아무리 웹 서버(80번 포트 등)를 돌려도, 외부(호스트 PC나 인터넷)에서는 그 입구를 찾을 수 없습니다. 이때 **호스트의 특정 문(포트)과 컨테이너의 문을 연결**해주는 것이 포트 포워딩입니다.

### 💡 동작 원리
* **NAT(Network Address Translation) 기술:** 도커 엔진은 호스트의 네트워크 인터페이스에 특정 포트를 점유하고, 해당 포트로 들어오는 모든 트래픽을 컨테이너의 내부 IP와 포트로 전달(Routing)합니다.
* **다리 역할:** 외부 사용자가 `호스트IP:8080`으로 접속하면, 도커가 이를 가로채서 `컨테이너내부IP:80`으로 쏘아주는 방식입니다.

### 🛠 명령어 구성
```bash
docker run -p [호스트_포트]:[컨테이너_포트] [이미지_이름]
```
* **예시:** `docker run -p 8080:80 nginx`
    * **호스트 포트 (8080):** 내 컴퓨터에서 접속할 번호
    * **컨테이너 포트 (80):** 이미지 내부에서 돌아가는 서비스 번호

## 2. 바인드 마운트 (Bind Mount, `-v`)

컨테이너는 '수정 불가능한 상태'로 유지되는 것이 원칙입니다. 컨테이너가 삭제되면 내부 데이터도 사라지죠(Stateless). 이를 해결하기 위해 **호스트 컴퓨터의 특정 폴더를 컨테이너 내부 폴더와 실시간으로 동기화**하는 것이 바인드 마운트입니다.

### 💡 동작 원리
* **파일 시스템 연결:** 호스트의 경로를 컨테이너 내부의 경로에 "덮어쓰기" 혹은 "연결"합니다. 
* **실시간 공유:** 호스트에서 파일을 수정하면 컨테이너에 즉시 반영되고, 반대로 컨테이너가 파일을 생성하면 호스트에도 남습니다. 
* **영속성 보장:** 컨테이너를 삭제하고 새로 만들어도, 데이터는 호스트에 남아있으므로 안전합니다.

### 🛠 명령어 구성
```bash
docker run -v [호스트_절대_경로]:[컨테이너_내부_경로] [이미지_이름]
```
* **예시:** `docker run -v /Users/me/project:/usr/share/nginx/html nginx`
    * **호스트 경로:** 반드시 **절대 경로**를 사용해야 합니다.
    * **컨테이너 경로:** 해당 데이터를 받아올 컨테이너 안의 위치입니다.

---

## 3. 한눈에 비교하기

| 구분 | 포트 포워딩 (`-p`) | 바인드 마운트 (`-v`) |
| :--- | :--- | :--- |
| **목적** | **네트워크** 통로 개방 | **데이터** 영구 저장 및 공유 |
| **대상** | TCP/UDP 포트 번호 | 파일 시스템 디렉토리/파일 |
| **핵심 이점** | 외부에서 컨테이너 서비스 접속 가능 | 컨테이너 삭제 후에도 데이터 보존 |
| **실생활 비유** | 아파트 입구(호스트)에서 특정 호수(컨테이너)로 연결되는 인터폰 | 내 방 책상 서랍(호스트)을 옆 방(컨테이너) 벽에 뚫어서 같이 쓰기 |

---

> **[Tip]** 최근에는 `-v` 대신 더 명확한 표현인 `--mount` 옵션을 권장하기도 합니다. 
> 예: `--mount type=bind,source=/host/path,target=/container/path` 
> 하지만 직관적이고 짧은 `-v`가 여전히 실무에서 압도적으로 많이 쓰입니다.

혹시 지금 진행 중인 프로젝트에서 특정 환경(예: DB 설정이나 소스 코드 실시간 반영)을 구축하려고 하시나요? 상황을 알려주시면 더 구체적인 설정법을 제안해 드릴 수 있습니다.

## 4 docker network
첨부해주신 내용을 바탕으로 도커 컨테이너의 **가상 주소(vNIC)**와 호스트를 연결하는 **브리지(Bridge)**의 개념 및 작동 원리를 정리해 드립니다.

### 1. 컨테이너의 가상 주소와 가상 NIC (vNIC)
도커 컨테이너는 실제 물리적인 랜카드(NIC)가 없지만, 소프트웨어로 구현된 **가상 NIC(vNIC)**을 통해 독립적인 네트워크 주소를 보유합니다.

*   **네트워크 네임스페이스 (Network Namespace):** 리눅스 커널은 하나의 OS 안에 독립된 네트워크 환경(방)을 여러 개 만들 수 있는데, 이를 네임스페이스라고 합니다. 각 컨테이너는 자신만의 방을 가지며 그 안에서 **독자적인 IP 주소, 라우팅 테이블, 방화벽 규칙**을 관리합니다.
*   **가상 이더넷 쌍 (veth pair):** 진짜 랜선 대신 **veth**라는 가상 장치를 사용합니다. 이는 양쪽에 플러그가 달린 랜선과 같아서, 한쪽 끝(eth0)은 컨테이너 내부에 꽂혀 **가상 NIC** 역할을 하고, 다른 한쪽 끝(veth...)은 호스트의 브리지에 연결됩니다.
*   **독립적인 정체성:** 컨테이너 내부에서 `hostname -I` 명령어를 치면 나오는 주소(예: 192.168.215.2)가 바로 컨테이너의 고유한 이름표이자 정체성입니다.

### 2. 가상 브리지 (docker0 / bridge101)
호스트 OS에 존재하는 **브리지**는 컨테이너 세상과 호스트 세상을 연결해 주는 **가상 L2 스위치**이자 **복도의 관리소(입구)** 역할을 합니다.

*   **연결 중심점:** 모든 컨테이너의 가상 랜선(veth)들이 이 브리지 스위치에 모여 연결됩니다. 이 덕분에 컨테이너끼리 서로 통신할 수 있습니다.
*   **게이트웨이 (Gateway) 역할:** 브리지는 자신만의 IP(예: 192.168.215.1)를 가집니다. 이는 컨테이너들이 바깥세상(인터넷)으로 나갈 때 반드시 통과해야 하는 **'대문'** 주소가 됩니다.
*   **라우팅 통로:** 호스트에서 컨테이너 주소로 핑(ping)을 날리면, 호스트의 라우팅 테이블이 "이 주소는 브리지가 관리하는 곳이다"라고 판단하여 브리지로 데이터를 보내고, 브리지가 해당 컨테이너로 전달해 줍니다.

### 3. 작동 원리 요약 (아파트 비유)
첨부된 소스에서는 이를 이해하기 쉽게 **아파트**에 비유하고 있습니다.

*   **호스트 컴퓨터:** 아파트 전체 건물 주소 (예: 서울시 강남구 도커 아파트).
*   **컨테이너:** 아파트 내부의 **독립된 방(101호, 102호)**. 각 방은 자기만의 전용 우편함(사설 IP)을 가집니다.
*   **브리지:** 각 방으로 연결된 복도의 **관리소 또는 공용 현관**. 외부 택배(데이터)가 오면 각 방으로 전달해 주는 통로입니다.
*   **컨테이너 삭제:** 컨테이너를 삭제(`rm -f`)하면 101호라는 공간 자체가 사라지는 것과 같아, 그 주소로 편지를 보내도 응답을 받을 수 없게(Request timeout) 됩니다.

### 4. 환경에 따른 차이 (macOS)
macOS 사용자의 경우, 도커가 가벼운 **가상 머신(VM)** 안에서 실행되므로 호스트의 `ifconfig` 명령어로는 컨테이너의 가상 NIC이 직접 보이지 않을 수 있습니다. 대신 `bridge101`과 같은 인터페이스가 호스트와 VM 내부의 컨테이너 네트워크를 이어주는 통로 역할을 수행하는 것을 확인할 수 있습니다.

### docker attach exec
도커를 공부하다 보면 가장 헷갈리는 두 명령어가 바로 `docker exec`와 `docker attach`입니다. 둘 다 "실행 중인 컨테이너에 접속한다"는 느낌은 비슷하지만, 결정적인 차이가 있습니다.

아주 쉬운 비유로 설명해 드릴게요.

---

### 1. 한 줄 요약 비유
* **`docker attach`**: 이미 거실에서 TV를 보고 있는 사람 옆에 **'나란히 앉아서'** 같은 화면을 보는 것 (화면 공유)
* **`docker exec`**: 거실에서 누가 뭘 하든 상관없이, 나는 **'뒷문으로 몰래 들어가서'** 나만의 일을 따로 하는 것 (새로운 작업 시작)

---

### 2. 상세 설명

#### ① docker attach (기존 프로세스에 합류)
컨테이너가 처음 실행될 때 돌아가고 있던 **메인 프로세스(1번 프로세스)**의 터미널에 그대로 접속합니다.
* **특징**: 내가 입력하는 것이 컨테이너의 메인 프로세스에 그대로 전달됩니다.
* **주의사항**: 만약 `attach` 상태에서 `Ctrl + C`를 눌러서 빠져나오려고 하면, **메인 프로세스가 종료되면서 컨테이너 자체가 꺼져버립니다.** (나오고 싶을 때는 `Ctrl + P, Q`를 순서대로 눌러야 컨테이너를 살린 채 탈출할 수 있습니다.)
* **용도**: 컨테이너가 지금 무슨 일을 하고 있는지 실시간 로그를 확인하거나, 실행 중인 프로그램과 직접 상호작용해야 할 때 씁니다.

#### ② docker exec (새로운 프로세스 실행)
이미 실행 중인 컨테이너 안에 **새로운 명령어(프로세스)를 하나 더** 실행합니다.
* **특징**: 메인 프로세스와는 별개로 움직입니다. 보통 `-it /bin/bash`를 붙여서 컨테이너 내부의 쉘(Shell)을 새로 하나 띄우는 용도로 가장 많이 씁니다.
* **장점**: 여기서 `exit`를 치거나 `Ctrl + C`를 눌러도 내가 새로 만든 쉘만 꺼질 뿐, **원래 돌아가던 컨테이너(메인 프로세스)는 영향을 받지 않고 계속 돌아갑니다.**
* **용도**: 컨테이너 내부의 파일을 수정하거나, 설정값을 확인하거나, 디버깅을 위해 임시로 들어갈 때 가장 안전하고 흔하게 사용합니다.

---

### 3. 표로 비교하기

| 구분 | docker attach | docker exec |
| :--- | :--- | :--- |
| **대상** | 컨테이너의 **메인 프로세스** | 컨테이너 내 **새로운 프로세스** |
| **화면** | 실행 중인 로그/화면을 공유함 | 새로운 터미널 창을 여는 느낌 |
| **종료 시 영향** | `Ctrl+C` 시 **컨테이너 종료** | `exit` 시 **컨테이너 유지** |
| **주요 용도** | 실행 중인 서비스 모니터링 | 내부 점검, 파일 수정, 디버깅 |

---

### 💡 실습 팁!
보통 실무에서는 컨테이너를 죽일 위험이 없는 **`docker exec -it [이름] bash`** 명령어를 90% 이상 사용합니다. 

`attach`는 마치 TV 앞에 같이 앉는 것과 같아서, 내가 리모컨(터미널 입력)을 잘못 만지면 TV가 꺼질 수 있다는 점만 기억하시면 됩니다! 조금 더 명확해지셨나요?

결론부터 말씀드리면, **`docker run attach`나 `docker run exec`라는 명령어는 존재하지 않습니다.** 도커 명령어는 **[동사] + [대상]**의 구조를 가지는데, `run`, `attach`, `exec`는 각각 서로 다른 동작을 하는 독립적인 **'동사'**들이기 때문입니다.

왜 섞어서 쓸 수 없는지, 그리고 비슷한 효과를 내려면 어떻게 해야 하는지 쉽게 알려드릴게요.

---

### 1. 왜 안 되나요? (역할의 중복)

* **`run`**: "방을 새로 만들고 사람을 들여보낸다."
* **`attach`**: "이미 사람이 있는 방에 나란히 앉는다."
* **`exec`**: "이미 사람이 있는 방에 뒷문으로 새 사람을 보낸다."

`run`은 **생성**이 목적이고, `attach/exec`는 **이미 생성된 것**에 대한 동작이라서 한 문장에 쓸 수 없습니다. 

---

### 2. "만들자마자 바로 접속"하고 싶을 때 (대안)

사용자님이 `docker run attach`를 생각하신 이유는 **"컨테이너를 만들자마자 바로 그 안의 터미널을 쓰고 싶어서"**일 거예요. 그럴 때는 `attach`를 붙이는 게 아니라 **옵션(`-it`)**을 사용합니다.

#### ✅ `run` + `attach` 같은 효과를 내려면:
```bash
docker run -it ubuntu bash
```
* **`-i` (interactive)**: 표준 입력을 열어둠 (내가 치는 글자가 전달됨)
* **`-t` (tty)**: 가상 터미널을 할당함 (화면이 예쁘게 나옴)
* 이렇게 하면 컨테이너가 **생성됨과 동시에** 내부 터미널로 쏙 들어갑니다. (이게 사실상 `attach`된 상태와 같습니다.)

#### ✅ 컨테이너를 실행만 해두고 나중에 들어가려면:
1.  일단 실행 (데몬 모드 `-d`):
    `docker run -d --name my_server nginx`
2.  나중에 필요할 때 접속 (`exec` 권장):
    `docker exec -it my_server bash`

---

### 3. 한눈에 보는 명령어 조합법

| 내가 원하는 것 | 올바른 명령어 |
| :--- | :--- |
| **새로 만들어서 바로 들어가기** | `docker run -it [이미지] [명령어]` |
| **이미 돌아가는 곳에 합류하기** | `docker attach [컨테이너]` |
| **이미 돌아가는 곳에 일 보러 가기** | `docker exec -it [컨테이너] [명령어]` |

---

### 💡 요약
* `docker run attach` (X) → **`docker run -it`** (O)
* `docker run exec` (X) → **`docker run`으로 먼저 만든 후 `docker exec`** (O)

도커는 한 번에 한 가지 동작만 명확히 지시하는 것을 좋아합니다! 이제 `run -it`만 기억하셔도 웬만한 실습은 문제없으실 거예요. :)

mpeg46551@c5r1s2 Codyssey % docker exec -it affectionate_raman bash
root@8ede664b8fb5:/# ls 
bin  boot  dev  etc  home  lib  lib64  media  mnt  opt  proc  root  run  sbin  srv  sys  tmp  usr  var
root@8ede664b8fb5:/# exit
exit
mpeg46551@c5r1s2 Codyssey % docker ps
CONTAINER ID   IMAGE     COMMAND   CREATED          STATUS          PORTS     NAMES
8ede664b8fb5   ubuntu    "bash"    28 minutes ago   Up 28 minutes             affectionate_raman

네, 맞습니다! `docker exec`를 쓸 때 마지막에 `bash`를 붙이는 이유를 정확히 포착하셨네요.

그 이유를 아주 쉽게 설명해 드릴게요.

### 1. `exec`는 "무슨 일을 할지"를 기다리고 있어요
`docker exec`의 기본 구조는 이렇습니다.
> **docker exec [옵션] [컨테이너이름] [실행할 명령]**

여기서 마지막에 넣는 **`bash`**가 바로 **"실행할 명령"**입니다.
* 만약 `bash`를 안 쓰면, 도커는 "컨테이너에 들어는 왔는데... 여기서 뭘 할까? 파일을 보여줄까(ls)? 네트워크를 볼까(ifconfig)?" 하고 망설이게 됩니다.
* 그래서 우리는 **"야, 나랑 대화할 수 있게 '터미널 프로그램(bash)'을 실행해줘!"**라고 명확히 시키는 것이죠.

### 2. `bash` 대신 다른 걸 넣으면 어떻게 될까요?
꼭 `bash`만 써야 하는 건 아닙니다. 방에 들어가지 않고 **밖에서 심부름만 시킬 수도 있어요.**

* **안 들어가고 파일 목록만 보기:**
  `docker exec affectionate_raman ls`
  (이렇게 치면 `ls` 결과만 딱 보여주고 바로 내 터미널로 돌아옵니다.)

* **안 들어가고 IP 주소만 확인하기:**
  `docker exec affectionate_raman hostname -I`

### 3. `-it`와 `bash`의 환상적인 콤비
사용자님이 쓰신 명령어 세트를 분석하면 이렇습니다:
* **`-i`**: 내가 키보드로 치는 걸 컨테이너에 전달해줘!
* **`-t`**: 터미널 화면처럼 예쁘게 꾸며줘!
* **`bash`**: 대화형 쉘 프로그램을 실행해줘!

이 세 개가 합쳐져야 비로소 우리가 흔히 아는 **"컨테이너 안으로 들어가서 직접 조작하기"**가 가능해지는 것입니다.

### 💡 팁: 이미지에 따라 `bash`가 없을 때도 있어요!
어떤 가벼운 이미지(Alpine Linux 등)는 `bash`가 아예 설치 안 된 경우도 있습니다. 그럴 때는 당황하지 마시고 **`sh`**를 써보세요.
> `docker exec -it [이름] sh`

방금 직접 `ls`로 내부 폴더들을 확인하고 `exit`로 안전하게 나오신 과정이 아주 정석적인 도커 활용법입니다! 이제 컨테이너 내부를 들여다보는 데 도사가 되셨네요. :)
