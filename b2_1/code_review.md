# 가계부 애플리케이션 (`budget_app`) 코드 리뷰 및 구조 분석 보고서

현재 구현된 `b2_1/budget_app` 코드는 **파이썬 표준 라이브러리만을 사용하여 고도로 최적화되고 안전하게 작성된 매우 모범적인 CLI 애플리케이션**입니다. 디자인 패턴, 데이터 정합성, 메모리 최적화 등 공부용 및 템플릿용으로 픽스하여 활용하기에 **더할 나위 없이 훌륭한 코드**입니다.

이 코드가 왜 잘 작성된 코드인지, 구조적 강점과 약간의 개선 포인트(리팩토링 제안)를 상세히 정리해 드립니다.

---

## 1. 핵심 아키텍처 및 설계적 강점 (공부할 만한 핵심 포인트)

### ① 레이어드 아키텍처 (Layered Architecture) 및 의존성 주입 (DI)
* **구조**: `Models (데이터 구조)` ➔ `Repository (영속성/파일 CRUD)` ➔ `Service (비즈니스 로직)` ➔ `CLI (사용자 인터페이스)` 순으로 계층화가 매우 뚜렷하게 분리되어 있습니다.
* **진입점**: `__main__.py`에서 파일 저장 경로를 전달받아 `FileRepository`를 생성하고, 이를 `BudgetService`에 주입(Injection)한 뒤 최종적으로 `InteractiveShell`을 띄웁니다.
* **장점**: 결합도가 낮아 각 클래스에 대한 단위 테스트(`unittest`)를 작성하기 매우 용이합니다.

### ② 메모리 효율적인 제너레이터 기반 파일 스트리밍
* 일반적인 파일 가계부는 파일 전체를 메모리에 한 번에 올리는(`json.load` 등) 실수를 범하기 쉽습니다.
* 본 프로젝트는 `yield` 제너레이터를 사용하여 JSONL 파일의 각 행을 **한 줄씩 읽어 들이는 스트리밍 방식**을 철저히 구현했습니다.
* 특히 조회(`list`, `search`) 시 전체 데이터를 메모리에 두지 않고, **최대 `limit` 크기로 제한된 정렬 삽입 버퍼(Sorted Insertion Buffer)**를 두어 메모리 복잡도를 $O(\text{limit})$으로 억제한 점은 메모리 관리 측면에서 극도로 훌륭한 예제입니다.

### ③ 데이터 정합성을 위한 원자적 파일 교체 (Atomic Write)
* 거래 삭제/수정, 예산 기입 등 데이터를 변경할 때 원본 파일에 직접 스트림 쓰기를 하다가 프로그램이 도중에 종료되면 파일 전체가 손상(Corrupted)됩니다.
* 이 코드는 이를 방지하기 위해 `tempfile` 모듈로 **임시 파일에 안전하게 모든 쓰기를 마친 뒤**, 최종 성공 시에만 OS 레벨의 원자적 교체 함수인 `os.replace`를 사용하여 안전하게 덮어씁니다.
* 시스템 중단이나 예기치 않은 단절 상황에서도 데이터 손상을 완벽하게 방어하는 실무급 기법입니다.

### ④ 데코레이터 패턴을 통한 관심사 분리 (AOP)
* 트래킹 로깅, 경과 시간 측정, 에러 가두기 등의 부가 기능이 핵심 비즈니스 로직 및 CLI 코드와 느슨하게 결합되어 있습니다.
* `@catch_errors`는 프로세스가 통째로 죽는 대신 에러 힌트를 출력하고 다시 셸 프롬프트 루프를 살려두도록 구현되어 있어 데코레이터의 훌륭한 활용 사례를 보여줍니다.

### ⑤ 표준 라이브러리만을 활용한 고기능 터미널 UI
* 외부 무거운 라이브러리(`click`, `prompt_toolkit` 등) 없이 파이썬 내장 `readline` 모듈만을 활용하여 **커맨드 자동완성, 화살표 키 탐색, 이전 입력 기록(History), 인라인 기본값 자동 채우기 및 백스페이스 덮어쓰기**를 구현했습니다.
* CJK(한글) 문자가 터미널에서 영문과 너비가 달라 테이블이 깨지는 문제를 보정하기 위해 직접 문자 너비 폭 계산기(`print_aligned_rows`)를 구현한 것도 대단히 완성도가 높습니다.

## 3. 리팩토링 및 개선 가능한 미세 포인트 (공부하며 다듬어 볼 만한 주제)

공부를 위해 코드를 더 정교하게 다듬어보고 싶으시다면, 아래의 미세 포인트를 개선해보는 것을 권장합니다.

### ① 반복 거래 템플릿(Recurring Templates)의 저장 및 로드 경로 리팩토링
* **현재 코드 상황**: 
  - `load_recurring_templates` 및 `save_recurring_templates` 메서드가 `repository.py`가 아닌 `service.py` 내부(347~375라인)에 정의되어 있습니다.
  - 이로 인해 `service.py`가 직접 `self.repository.recurring_path`와 `tempfile` 경로에 접근하여 파일 읽기/쓰기를 제어하고 있습니다.
* **개선 제안**:
  - 데이터 영속성 처리의 일관성을 위해 이 두 함수를 `repository.py`(`FileRepository` 클래스) 내부로 이관하고, `service.py`에서는 `self.repository.load_recurring_templates()` 형식으로 호출하도록 수정하면 완벽한 책임 분리(SRP)가 달성됩니다.

### ② CSV 임포트 시 인라인 유효성 검사 리팩토링
* **현재 코드 상황**:
  - `service.py`의 `import_from_csv` 메서드 내부(285~297라인)에 날짜, 타입, 카테고리 등을 체크하는 유효성 검사 로직이 하드코딩 형태로 인라인 기재되어 있습니다.
* **개선 제안**:
  - 이미 아래에 정의되어 있는 `self.validate_fields(date, type_str, category, amount)` 헬퍼 메서드를 재활용하여 검증하도록 통합하면 중복 코드가 제거되고 검증 로직이 일원화됩니다.
